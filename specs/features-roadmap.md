# UDFKit Features Roadmap

## Context

This document tracks new user-facing capabilities planned for UDFKit beyond the v2 quality improvements in `v2-improvement-plan.md`. Items are ordered by implementation dependency — earlier items unblock later ones. Each feature ships as its own PR.

---

## Feature 1 — `AsyncSequence` State Streaming

**Why:** SwiftUI apps observe state via `@Observable` / `@Published`. Non-SwiftUI consumers (CLI tools, background agents, unit tests) have no way to observe state changes without polling. An `AsyncSequence` interface makes UDFKit usable outside UIKit/SwiftUI.

**API:**

```swift
// Observation
for await state in store.stateStream {
    print("State changed: \(state)")
}

// One-shot next state
let nextState = await store.stateStream.first(where: { $0.count > 0 })
```

**Implementation sketch:**

```swift
// Store.swift
private var continuation: AsyncStream<State>.Continuation?

public var stateStream: AsyncStream<State> {
    AsyncStream { continuation in
        self.continuation = continuation
        continuation.yield(state)  // emit current state immediately
    }
}

// In dispatch, after applying reducer:
continuation?.yield(state)
```

**Considerations:**
- Only one `AsyncStream` at a time with this design; for multiple subscribers use `AsyncChannel` from `swift-async-algorithms` or a broadcast pattern with `[AsyncStream<State>.Continuation]`
- Stream terminates when `Store` deallocates (`continuation?.finish()` in `deinit`)
- Backpressure: `AsyncStream` buffers by default; high-frequency dispatches may queue up — document the limitation

**Files to create/modify:**
- `Sources/UDFKit/Core/Store.swift`
- `Tests/UDFKitTests/Core/StoreTests.swift` (add `stateStream` tests)

**Acceptance criteria:** `for await state in store.stateStream` compiles and emits a value on every `dispatch`.

## Feature 2 — Result Builder DSL for Store Configuration

**Why:** Registering effects and reducers requires repeated `store.builderEffects.registerEffect(...)` calls. A result builder DSL enables declarative, compile-time-safe store setup analogous to SwiftUI's `body`.

**API:**

```swift
let store = Store(state: AppState()) {
    CounterReducer()  // registered via ReducerBuilder
    ProfileReducer()
} effects: {
    CounterEffect()   // registered via EffectsBuilder
    AnalyticsEffect()
}
```

**Implementation sketch:**

```swift
@resultBuilder
public struct ReducerBuilder<State: StoreState, Action: StoreAction> {
    public static func buildBlock(_ reducers: any Reducer<State, Action>...) -> [any Reducer<State, Action>] {
        reducers
    }
}

@resultBuilder
public struct EffectsBuilder<State: StoreState, Action: StoreAction> {
    public static func buildBlock(_ effects: any Effect<State, Action>...) -> [any Effect<State, Action>] {
        effects
    }
}
```

New `Store` init overload accepting builder closures:

```swift
public init(
    state: State,
    @ReducerBuilder<State, Action> reducers: () -> [any Reducer<State, Action>],
    @EffectsBuilder<State, Action> effects: () -> [any Effect<State, Action>]
)
```

**Considerations:**
- The builder-based init supplements — not replaces — the existing init that accepts `BuilderReducer` and `BuilderEffects` directly
- Heterogeneous reducer/effect collections require type erasure (`any Reducer`, `any Effect`) or a concrete wrapper

**Files:**
- `Sources/UDFKit/Builders/ReducerBuilder.swift` (new)
- `Sources/UDFKit/Builders/EffectsBuilder.swift` (new)
- `Sources/UDFKit/Core/Store.swift` (new init overload)
- `Tests/UDFKitTests/Builders/` (new test files)

## Feature 4 — Middleware / Interceptor Layer

**Why:** Cross-cutting concerns (analytics, logging, crash reporting, undo tracking) currently require wrapping every reducer or writing a custom `Effect`. A `Middleware` protocol provides a dedicated interception point without modifying reducer or effect code.

**API:**

```swift
public protocol Middleware<State, Action>: Sendable {
    associatedtype State: StoreState
    associatedtype Action: StoreAction

    func intercept(
        state: State,
        action: Action,
        next: @Sendable (Action) async -> Void
    ) async
}
```

Usage:

```swift
let store = Store(
    state: AppState(),
    reducer: appReducer,
    effects: [appEffect],
    middleware: [LoggingMiddleware(), AnalyticsMiddleware()]
)
```

**Implementation sketch:**
- `Store.dispatch` builds a middleware chain: `[M1, M2, ..., Mn, finalReducer]`
- Each middleware calls `next` to pass through to the next middleware or the reducer
- Middlewares can short-circuit by not calling `next`, or transform the action

**Relationship to `LogsEffect`:**
- `LogsEffect` as a middleware is simpler (no `nil` return, direct interception)
- Keep `LogsEffect` as an `Effect` for backward compatibility; add `LogsMiddleware` as an alternative

**Files:**
- `Sources/UDFKit/Core/Middleware.swift` (new)
- `Sources/UDFKit/Core/Store.swift` (middleware chain in `dispatch`)
- `Sources/UDFKit/GenericEffects/LogsMiddleware.swift` (new)
- `Tests/UDFKitTests/Core/MiddlewareTests.swift` (new)

## Feature 5 — `@StoreReducer` Macro

**Why:** Large apps define many sub-reducers. Wiring them into a `BuilderReducer` requires repetitive `registerReducer(\.subState, SubReducer())` calls. The `@StoreReducer` macro can generate this boilerplate.

**API:**

```swift
@StoreReducer
struct AppReducer {
    let counter = CounterReducer()   // KeyPath \.counter inferred
    let profile = ProfileReducer()
}
```

Generates:

```swift
extension AppReducer: Reducer {
    func reduce(oldState: AppState, with action: AppAction) -> AppState {
        var state = oldState
        state = counter.reduce(oldState: state, with: action)
        state = profile.reduce(oldState: state, with: action)
        return state
    }
}
```

**Considerations:**
- Requires the macro to inspect property types and infer key paths — harder than `@StoreActionWrapper` (which only reads enum cases)
- State type must conform to `StoreState` and have matching sub-state properties
- Likely requires `MemberMacro` + `ExtensionMacro` combination

**Files:**
- `Sources/UDFKitMacros/StoreReducerMacro.swift` (new)
- `Sources/UDFKitMacros/Plugin.swift` (register new macro)
- `Sources/UDFKit/Core/StoreDefinitions.swift` (declare `@StoreReducer` macro)
- `Tests/UDFKitMacrosTests/StoreReducerMacroTests.swift` (new)

## Feature 6 — Undo / Redo Support

**Why:** Form-heavy or canvas-based apps need undo/redo. Implementing it per-app requires duplicating state history logic.

**API:**

```swift
let store = UndoableStore(
    state: EditorState(),
    reducer: editorReducer,
    effects: [],
    maxHistoryDepth: 50
)

await store.dispatch(.drawLine(...))
await store.undo()   // reverts to state before drawLine
await store.redo()   // re-applies drawLine
```

**Implementation sketch:**

```swift
@Observable
@MainActor
public final class UndoableStore<State: StoreState, Action: StoreAction>: Store<State, Action> {
    private var history: [State] = []
    private var redoStack: [State] = []
    private let maxDepth: Int

    override func applyReducer(oldState: State, action: Action) -> State {
        history.append(oldState)
        if history.count > maxDepth { history.removeFirst() }
        redoStack.removeAll()
        return super.applyReducer(oldState: oldState, action: action)
    }

    public func undo() {
        guard let previous = history.popLast() else { return }
        redoStack.append(state)
        state = previous
    }

    public func redo() {
        guard let next = redoStack.popLast() else { return }
        history.append(state)
        state = next
    }
}
```

**Considerations:**
- History stores full state copies — fine for small structs, expensive for large nested states
- Actions that trigger side effects (network calls) in effects are not undone — document this limitation clearly
- `canUndo: Bool` and `canRedo: Bool` computed properties for binding to button enabled state

**Files:**
- `Sources/UDFKit/Core/UndoableStore.swift` (new)
- `Tests/UDFKitTests/Core/UndoableStoreTests.swift` (new)

## Feature 7 — Persistence Effect

**Why:** Apps commonly need to persist state (user preferences, drafts). Writing a custom persistence effect is boilerplate-heavy.

**API:**

```swift
let persistenceEffect = PersistenceEffect<AppState>(
    key: "app_state",
    storage: .userDefaults,
    restoreOnInit: true
)

let store = Store(
    state: persistenceEffect.restoredState ?? AppState(),
    reducer: appReducer,
    effects: [persistenceEffect]
)
```

**Implementation sketch:**

```swift
public struct PersistenceEffect<State: StoreState & Codable>: Effect {
    public typealias Action = Never  // observes all actions, never emits one

    private let key: String
    private let encoder = JSONEncoder()
    private let decoder = JSONDecoder()

    public var restoredState: State? {
        guard let data = UserDefaults.standard.data(forKey: key) else { return nil }
        return try? decoder.decode(State.self, from: data)
    }

    public func process(state: State, with action: Never) async -> Never? { nil }

    // Override the state-only variant to persist after every dispatch
    public func process(state: State, with action: some StoreAction) async -> (some StoreAction)? {
        if let data = try? encoder.encode(state) {
            UserDefaults.standard.set(data, forKey: key)
        }
        return nil
    }
}
```

**Considerations:**
- `Action = Never` conflicts with the `Effect` protocol requirement — may need a separate `StateObserver` protocol
- Debounce writes for high-frequency dispatch (don't write to disk 1000×/sec)
- Support file-based storage as an alternative to `UserDefaults` for large state objects
- `State` must conform to `Codable` — document this constraint

**Files:**
- `Sources/UDFKit/GenericEffects/PersistenceEffect.swift` (new)
- `Tests/UDFKitTests/GenericEffects/PersistenceEffectTests.swift` (new)

## Implementation Priority

| Feature | Complexity | Value | Suggested Order |
|---------|-----------|-------|-----------------|
| 1 — AsyncSequence streaming | Low | High | First |
| 3 — DocC documentation | Low | High | Second (no deps) |
| 2 — Result builder DSL | Medium | Medium | Third |
| 4 — Middleware | Medium | High | Fourth |
| 5 — @StoreReducer macro | High | Medium | Fifth |
| 6 — Undo/Redo | Medium | Medium | Sixth |
| 7 — Persistence | Medium | High | Seventh |

Features 1 and 3 are independent of all others and provide immediate value. Feature 4 (Middleware) should precede Features 6 and 7, as both could be implemented as middlewares.

---

## Breaking Change Policy

Features 1–3 are additive and non-breaking.
Features 4–7 add new types and init overloads — also non-breaking if implemented correctly.

Any breaking change (removing `AnyEffect.createEffect`, changing `Store` init signature) must:
1. Be documented in `CHANGELOG.md` under a new `## [2.0.0]` section
2. Include a migration guide in `Documentation.docc/Articles/MigrationGuide.md`
3. Follow semantic versioning — bump major version in README and Package description
